================================================================================
  WORKLOAD-SPECIALIZED SSD-BACKED FILE FORMAT PLAN
  TPC-H Variant — 8 Queries over 7 Tables
================================================================================

────────────────────────────────────────────────────────────────────────────────
0.  WORKLOAD SUMMARY AND DESIGN PHILOSOPHY
────────────────────────────────────────────────────────────────────────────────

Queries at a glance
───────────────────
  Q1  lineitem   – full scan, l_shipdate <= cutoff, group by returnflag/status,
                   aggregates on qty/price/discount/tax
  Q2  orders     – o_orderdate in 3-month window, group by orderpriority
  Q3  customer   – c_mktsegment = X, project custkey/name/acctbal/phone
  Q4  part       – p_brand = X AND p_container = Y, project key/name/mfgr/price
  Q5  supplier   – s_comment LIKE '%word%', project all non-comment columns
  Q6  lineitem   – l_shipmode IN (2 values) AND l_shipdate >= cutoff, count+sum
  Q7  part       – p_name LIKE '%color%' AND p_type LIKE '%type', project 4 cols
  Q8  customer   – c_acctbal > 0 AND phone prefix IN (3 codes), project 3 cols

Dominant cost regime: dataset >> RAM  ->  bytes read from SSD is the single
biggest cost.  The format must minimize (a) columns read, (b) pages read within
a column, (c) bytes per value within a page.

Key observations driving design
────────────────────────────────
  * lineitem is the largest table (SF x 6 M rows), accessed by Q1 and Q6.
    Minimising lineitem bytes-read is the highest-value target.
  * orders is the second-largest (~1.5 M rows at SF=1), accessed only by Q2.
  * part (~200 k rows at SF=1), customer (~150 k), supplier (~10 k) are
    mid-size.  nation and region are tiny and not accessed.
  * All predicates are equality, range, IN-list, or LIKE.
    Equalities on low-cardinality columns  -> dictionary + bitset.
    Range on dates                         -> zone-map pruning on clustered data.
    LIKE '%...%' (infix)                   -> cannot prune by sort; minimize
                                             page size so unrelated cols are
                                             not co-located.
  * No query joins tables (all are single-table queries).
    Auxiliary join-key structures are not needed.
  * Aggregates need column data but never row identity ->
    late-materialisation of non-predicate columns is ideal.

Global file-format decisions
────────────────────────────
  FORMAT NAME: "CSF" (Columnar Segment File)

  * Fully columnar (DSM-style), one file per table.
  * Row-group ("segment") size = 131 072 rows (128 Ki rows).
    - Large enough to amortise per-segment metadata overhead.
    - Small enough that zone-map pruning skips meaningful fractions of data.
  * Within each segment, columns are stored as independently encoded column-
    blocks.  If a column-block compressed size exceeds 64 KiB it is split
    into sub-blocks of at most 64 KiB each (one SSD read unit).
  * Target page size: 64 KiB (matches SSD read unit, avoids partial waste).
  * Compression: LZ4 applied after encoding, per column-block.
    LZ4 decompresses at ~4 GB/s in software; overhead negligible vs I/O.
  * Nulls: separate 1-bit validity bitmap per column-block, stored only when
    the column is nullable; bit-packed to ceil(N/8) bytes.  Non-nullable
    columns omit the bitmap entirely (saves space and branch-prediction load).
  * Variable-length strings: dictionary-encoded when cardinality is low;
    raw strings stored as (offsets[]: uint32[], payload: bytes[]), both LZ4.
  * Metadata placement: FILE FOOTER at end of file (enables streaming writes).
    Footer contains: schema, segment count, per-segment per-column zone maps,
    per-segment byte offsets, global dictionaries, sparse clustering-key index,
    Bloom filters, and auxiliary lookup tables.
  * Writers stream segment by segment.  Global dictionaries require one
    pre-scan or two-pass write; bounded by cardinality x avg-string-length
    (always <= a few KB per column).
  * Reader contract: given projected columns + predicates, the reader
      1. reads the footer (once, kept resident in RAM),
      2. evaluates zone maps / bitsets / Bloom filters per segment to produce
         a sorted segment read list,
      3. reads only the selected column-blocks for selected segments (sequential
         within each segment, coarse seeks across pruned segments),
      4. decodes and evaluates remaining row-level predicates inside each block,
      5. assembles output (late materialization for non-predicate columns).

1.  TABLE: lineitem  (largest table -- highest design priority)
================================================================================

TPC-H scale characteristics (SF=1 reference; scales linearly):
  Rows ~= 6 000 000.  128 Ki-row segments -> 46 segments at SF=1.

------------------------------------------------------------
1a. Per-column physical type and encoding
------------------------------------------------------------

Column            SQL type       Observed (min, max, distinct)  Phys type   Encoding
--------------------------------------------------------------------------------------
l_orderkey        INTEGER        (1, 6M, ~1.5M)                 uint32      FOR + delta
                  After clustering by l_shipdate, rows with nearby ship dates
                  tend to share nearby order keys (correlated).  Use
                  frame-of-reference (base = segment min) then delta-encode
                  the differences; bit-pack deltas at ceil(log2(max_delta+1))
                  bits.  Expected bits/value ~= 20 after FOR+delta.
                  Decode: vectorized bit-unpack + prefix-sum; predicate eval
                  on decoded values if ever filtered.

l_partkey         INTEGER        (1, 200k, 200k)                uint32      FOR + bit-pack
                  Range [1, 200k] fits in 18 bits after FOR (base=1).
                  Bit-packed at 18 bits/value.  No sortedness assumed.
                  Decode: vectorized bit-unpack.

l_suppkey         INTEGER        (1, 10k, 10k)                  uint16      FOR + bit-pack
                  Range [1, 10k] fits in 14 bits.  Physical uint16 (<=65535).
                  Bit-packed at 14 bits/value.

l_linenumber      INTEGER        (1, 7, 7 distinct)              uint8       bit-pack (3 bits)
                  Only 7 distinct values.  3 bits/row; vectorized unpack.
                  Could use RLE but values are not sorted -> bit-pack cheaper.

l_quantity        DECIMAL(15,2)  (1.00-50.00, 50 distinct)      uint8       raw uint8
                  All values are integers at .00 cents (TPC-H spec).
                  Store as uint8 in [1,50].  1 byte/row.
                  Logical reader multiplies by 100 for decimal(15,2) context.

l_extendedprice   DECIMAL(15,2)  (900.00-104949.50, ~933k)      uint32      FOR + bit-pack
                  Scale x100 -> integers [90000, 10494950].  Range ~10.4M
                  -> 24 bits after FOR (base = segment min, ~4 bytes stored
                  once per segment).  Bit-packed at 24 bits/value.
                  Logical reader: (packed + base) / 100.0.

l_discount        DECIMAL(15,2)  (0.00-0.10, 11 distinct)       uint8       raw uint8
                  Values: 0,1,...,10 (in units of 0.01).  Store code 0-10.
                  1 byte/row.  Logical reader: code / 100.0.

l_tax             DECIMAL(15,2)  (0.00-0.08, 9 distinct)        uint8       raw uint8
                  Values: 0,1,...,8.  1 byte/row.  Same decode.

l_returnflag      CHAR(1)        ('A','N','R', 3 distinct)       uint2       bit-pack (2 bits)
                  Encoded as 0='A', 1='N', 2='R'.  2 bits/row.
                  Global dict (3 bytes) stored in footer.
                  Q1 groups by this column.  Predicate evaluation on codes
                  directly (dictionary comparison): no string decode needed.
                  Per-segment present_bitset (3 bits) for metadata.

l_linestatus      CHAR(1)        ('F','O', 2 distinct)           uint1       bit-pack (1 bit)
                  1 bit/row.  Global dict (2 bytes).
                  Q1 groups by this column.  Same predicate strategy.

l_shipdate        DATE           (1992-01-02 to 1998-12-01)      uint16      delta (days)
                  Days since epoch 1992-01-01.  Range = 2557 days -> fits
                  uint12 but stored as uint16 for alignment.
                  After clustering sort by l_shipdate, values within each
                  segment are quasi-monotone -> delta encode:
                    segment base = min date (uint16, once per segment)
                    per-value delta = bit-packed at ceil(log2(seg_range)) bits
                  Expected seg_range ~= 56 days (2557/46) -> 6 bits/row.
                  Zone map (min, max) per segment enables strong pruning for
                  Q1 (upper bound) and Q6 (lower bound).
                  Decode: bit-unpack + add base; predicate eval on decoded days.

l_commitdate      DATE           (1992-01-31 to 1998-10-31)      uint16      FOR + bit-pack
                  Same epoch.  Not sorted after clustering by shipdate.
                  FOR (base = seg min) + 11-bit pack (range <= 2400 days).

l_receiptdate     DATE           (1992-01-03 to 1998-12-31)      uint16      FOR + bit-pack
                  Same treatment as commitdate.  11-bit pack.

l_shipinstruct    CHAR(25)       (4 distinct values)             uint2       global dict
                  Global dictionary of 4 entries (~100 bytes total).
                  Per-row: 2 bits.  Dictionary is RAM-resident (in footer).
                  No per-segment dict needed.

l_shipmode        CHAR(10)       (7 distinct values)             uint3       global dict
                  Global dictionary of 7 entries (~70 bytes total).
                  Per-row: 3 bits.
                  Q6 predicate: IN (2 of 7 values) -> compare 3-bit code
                  against a 7-bit inclusion mask; no string decode.
                  Per-segment: present_bitset uint8 (7 bits used) ->
                  if neither target code bit is set, skip segment entirely.

l_comment         VARCHAR(44)    (high cardinality, ~unique)     raw+LZ4     offset+payload
                  Not projected by Q1 or Q6.  Late materialization: column-
                  block for l_comment is never read by Q1 or Q6.
                  Stored as uint32 offset array + LZ4-compressed byte payload.

------------------------------------------------------------
1b. Clustering key
------------------------------------------------------------
  CLUSTER BY: l_shipdate  (ascending)

  Justification:
  * Q1 predicate: l_shipdate <= DATE - DELTA (upper-bound range).
    Clustering ascending means all qualifying rows precede non-qualifying ones.
    For typical DELTA ~= 90 days: cutoff ~= 1998-09-02; rows with
    l_shipdate > cutoff are in the last 90/2557 ~= 3.5% of the date range,
    concentrated in the last ~2 segments.  Those segments are pruned by
    zone_min > cutoff check.  Minor absolute saving (~2 segments), but the
    compact delta encoding of l_shipdate (6 bits vs 16 bits unsorted) is the
    main win: saves 10 bits x 600M rows (SF=100) = 750 MB disk.
  * Q6 predicate: l_shipdate >= DATE (lower-bound range).
    With ascending clustering, all segments where zone_max < cutoff are pruned.
    For cutoff = 1994-01-01: (1994-1992)/(1998-1992) ~= 33% of date range
    -> ~33% of segments pruned (1510 of 4578 at SF=100).
    This is the dominant pruning win for Q6.
  * Weighted byte savings across workload:
    - Q1: delta encoding saves ~750 MB + 2-seg pruning saves ~15 MB = ~765 MB.
    - Q6: date pruning saves 33% of 855 MB baseline read ~= 425 MB.
    Total lineitem clustering benefit: ~1.2 GB saved at SF=100.

  External sort strategy (streaming-compatible):
    1. Ingest rows into fixed-size in-memory run buffers (~64 MB each).
    2. Sort each run buffer by l_shipdate; write as a sorted temp file.
    3. External k-way merge-sort temp files -> produce globally sorted stream.
    4. Write each 131 072-row sorted block as one CSF segment.
    Only the sort keys (l_shipdate + row pointers) need to be in memory
    during merge.  All other columns are read/written by row-id reference.

------------------------------------------------------------
1c. Page/block layout (per segment, per column)
------------------------------------------------------------
  Each segment = 131 072 rows.
  Each column in a segment is one "column-block" unless compressed size
  exceeds 64 KiB, in which case it is split into sub-blocks of 64 KiB.

  Column-block header (16 bytes, uncompressed):
    magic[2]            0xC5F0
    encoding_type[1]    enum: 0=RAW, 1=FOR_BITPACK, 2=DELTA_BITPACK,
                              3=GLOBAL_DICT_BITPACK, 4=RLE, 5=RAW_LZ4_STRING
    bit_width[1]        bits per value for bit-packed encodings (0 if N/A)
    compressed_bytes[4] size of LZ4-compressed payload
    raw_bytes[4]        size of decoded payload (for buffer allocation)
    num_values[4]       number of logical values (rows) in this block

  Followed by encoding-specific header:
    FOR_BITPACK:   base_value[4 or 8]
    DELTA_BITPACK: first_value[4], base_delta[4] (usually 0)
    GLOBAL_DICT_BITPACK: dict_id[2] (references footer dict table)
    RLE: (value, count) pairs inline in payload

  Then: LZ4-compressed payload bytes.

  Null representation:
    lineitem has no nullable columns -> no validity bitmaps needed.
    (Saves ceil(131072/8) = 16 KB of bitmap overhead per segment that would
    otherwise be required.)

  Column layout within a segment (fixed schema order):
    [l_orderkey block] [l_partkey block] ... [l_comment block]
  Columns are contiguous within a segment.  Each segment's absolute byte
  offset is stored in the footer segment-offset table.
  Within a segment, each column-block's start offset is derivable by
  scanning the segment sequentially or by the per-segment per-column
  offset sub-table in the footer.

------------------------------------------------------------
1d. Per-segment pruning metadata (zone maps, stored in footer)
------------------------------------------------------------
  For each segment, the footer stores a zone-map record:
    l_shipdate   : (min uint16, max uint16)  -- primary pruning key
    l_returnflag : present_bitset uint8 (3 bits used)  -- Q1 grouping
    l_linestatus : present_bitset uint8 (2 bits used)  -- Q1 grouping
    l_shipmode   : present_bitset uint8 (7 bits used)  -- Q6 IN pruning
    l_discount   : (min uint8, max uint8)
    l_quantity   : (min uint8, max uint8)

  Zone-map size (SF=100, 4578 segments):
    4578 segs x 12 bytes/seg = 55 KB.  Entirely RAM-resident.

  Pruning strategy:
    Q1: skip segment if zone_min(shipdate) > cutoff.
    Q6: skip segment if zone_max(shipdate) < cutoff.
        Additionally: skip if (shipmode_bitset & target_mask) == 0.

------------------------------------------------------------
1e. Auxiliary structures
------------------------------------------------------------
  Global dictionaries (in footer, always RAM-resident):
    l_shipinstruct : 4 entries x 25 chars = 100 bytes
    l_shipmode     : 7 entries x 10 chars =  70 bytes
    l_returnflag   : 3 entries x  1 char  =   3 bytes
    l_linestatus   : 2 entries x  1 char  =   2 bytes
    Total: ~175 bytes

  Sparse clustering-key index (in footer):
    One entry per segment: (seg_id uint16, min_shipdate uint16,
    max_shipdate uint16, file_byte_offset uint64) = 12 bytes/entry.
    4578 entries x 12 bytes = 55 KB.  RAM-resident (same as zone maps).

  Per-segment per-column offset table (in footer):
    4578 segments x 16 columns x 8 bytes (uint64 offset) = 586 KB.

  Footer total (lineitem, SF=100): ~700 KB.  Entirely RAM-resident.

------------------------------------------------------------
2b. Clustering key
------------------------------------------------------------
  CLUSTER BY: o_orderdate  (ascending)

  Justification:
  * Q2 predicate: o_orderdate >= DATE AND o_orderdate < DATE + 3 months.
    A 3-month window in a 6.6-year range is ~3/79 ~= 3.8% of the date span.
    With ascending clustering, the 3-month window is a contiguous block in
    the file.  Zone-map pruning: skip segments where zone_max < window_start
    or zone_min > window_end.
  * At SF=100 (1145 segments covering 2404 days), a 90-day window spans
    ~90/2404 x 1145 ~= 43 segments.  Segments read: ~44 (including 2 boundary
    segments).  Pruned: 1145 - 44 = 1101 segments (96% pruned).
  * Q2 reads only o_orderdate (predicate) and o_orderpriority (group-by).
    Both are very narrow (3 bits + 3 bits = 0.75 bytes/row).
    Bytes read: 44 segs x 131072 rows x 0.75 B ~= 4.3 MB.  Extremely fast.

  External sort: single-key sort on o_orderdate (uint16).
  Keys are tiny; sort is very fast even for 150M rows (radix sort ~= 2 GB/s).

------------------------------------------------------------

  Per segment zone-map record (in footer):
    o_orderdate     : (min uint16, max uint16)  -- primary Q2 pruning
    o_orderpriority : present_bitset uint8 (5 bits)

  Zone map size (SF=100): 1145 segs x 6 bytes = 6.9 KB.  RAM-resident.

2d. Auxiliary structures

    o_orderstatus   :   3 x  1 =     3 bytes

  Per-segment per-column offset table: 1145 x 9 x 8 = 82 KB.
  Sparse index: 1145 x 12 bytes = 13.7 KB.
  Footer total (orders, SF=100): ~112 KB.  Entirely RAM-resident.

------------------------------------------------------------
3b. Clustering key
------------------------------------------------------------
  CLUSTER BY: (c_mktsegment, -c_acctbal)
  Primary key: c_mktsegment (encoded uint3, 5 values).
  Secondary key: c_acctbal descending.

  Justification:
  * c_mktsegment has 5 distinct values.  Clustering by it groups all rows of
    each market segment together into consecutive segments in the file.
    At SF=100 (115 segments / 5 mktsegments = 23 segments per group):
    Q3 equality predicate on mktsegment prunes 4/5 of segments = 80% pruned.
    Q3 reads only the 23 segments for the target mktsegment.
  * Secondary sort by c_acctbal descending within each mktsegment group:
    - Q3 ORDER BY c_acctbal DESC: rows are already in output order within
      each segment group; a merge-sort across 23 segments produces the final
      output with minimal overhead.
    - Q8 predicate c_acctbal > 0: with descending sort, segments at the
      START of each mktsegment block have the highest acctbal values.
      The zone_max(acctbal) of the last segments in each block will be <= 0
      -> those segments are pruned.  TPC-H has ~50% negative balances
      -> roughly half the segments per mktsegment group are pruned.
      For Q8 (no mktsegment filter): all 115 segments considered, but ~50%
      are pruned by acctbal zone map -> ~58 segments read.
  * Weighted savings:
    - Q3: 80% of customer bytes saved (23 vs 115 segments).
    - Q8: ~50% of customer bytes saved (acctbal zone map prunes low-acctbal
      segments across all mktsegment groups).

  External sort: two-pass sort on (c_mktsegment_code ASC, c_acctbal DESC).
  Both are small integers -> radix sort is efficient.

------------------------------------------------------------
3c. Per-segment pruning metadata
------------------------------------------------------------
  Per segment zone-map record (in footer):
    c_mktsegment  : present_bitset uint8 (5 bits)  -- Q3 primary pruning
    c_acctbal     : (min int32, max int32)          -- Q8 acctbal > 0 pruning
    c_phone_cc    : present_bitset uint32 (25 bits) -- Q8 IN-list pruning

  Zone map size (SF=100): 115 segs x 14 bytes = 1.6 KB.  RAM-resident.

------------------------------------------------------------

------------------------------------------------------------
4b. Clustering key
------------------------------------------------------------
  CLUSTER BY: (p_brand ASC, p_container ASC)
  Packed as a composite uint11 sort key (5-bit brand + 6-bit container).

  Justification:
  * Q4: predicate p_brand = X AND p_container = Y.  Both are equality on
    low-cardinality columns (25 x 40 = 1000 combinations).  Clustering by
    (brand, container) ensures rows sharing a (brand, container) pair are
    stored consecutively.  At SF=100 (20M rows / 1000 combos = 20K rows/combo):
    each combo spans 20K / 131072 ~= 0.15 segments -> each combo fits in
    exactly 1 segment (on average, with possible overflow to 2 consecutive).
    Zone-map pruning: per-segment brand_bitset and container_bitset.
    A segment not containing the target brand code -> skip (5-bit mask check).
    A segment not containing the target container code -> skip.
    With clustering, only 1-2 segments out of 153 contain a given (brand,
    container) pair -> pruning ratio ~= 99%.
  * Q7: p_name LIKE '%COLOR%' cannot prune by (brand,container) clustering.
    Bloom filter on name tokens is used instead.  (Brand,container) clustering
    does not hurt Q7 since all segments must be checked anyway.

  External sort: single-pass sort on composite uint11 key.  At SF=100,
  20M x 4 bytes sort key = 80 MB -> fits in RAM for single-pass sort.

                 Avg ~62 B raw -> LZ4 ~44 B.

                 comment strings.  At query time, extract all trigrams of the

                 no comment in the segment can contain the word -> skip segment.
                 Trigram Bloom is probabilistic but false positives only cause
                 unnecessary reads, never incorrect results.
                 At SF=100 (8 segments, 125K comments/segment with avg 62 chars
                 each), all common English trigrams will appear in every segment
                 -> Bloom filter prunes 0 segments for common words.  For rare
                 words (specific TPC-H complaint text), may prune 1-2 segments.
                 The filter costs 256 bytes/segment = 2 KB total: negligible.

  SF=1000 (60B lineitem rows, 45780 segments):
    lineitem footer: ~6.96 MB
    All other tables: ~0.17 MB
    Total resident: ~7.1 MB
  Still trivially small relative to any modern server RAM budget.
  The remaining RAM (GBs to TBs) is available as OS page cache for hot
  column-blocks, effectively creating a transparent in-memory tier.

Paged structures (read on demand, NOT pinned):
  * Base column-block pages (64 KB each): read per-query, LRU-cached by OS.
  * Large string payload blocks (l_comment, p_name, s_comment, etc.).

9.  PHYSICAL FILE ORGANIZATION (CSF FORMAT SPECIFICATION)

File layout (one .csf file per table):

  Offset 0:
  ┌─────────────────────────────────────────┐
  │  FILE HEADER  (64 bytes, uncompressed)  │
  │  magic[8]          = "CSFv0001"         │
  │  schema_crc32[4]   (schema hash)        │
  │  num_columns[4]                         │
  │  num_segments[8]                        │
  │  segment_size[4]   = 131072 (rows)      │
  │  clustering_col[2] = column index (-1   │
  │                       if unsorted)      │
  │  clustering_dir[1] = 0:asc, 1:desc      │
  │  flags[1]          (bit 0: has_nulls)   │
  │  footer_offset[8]  (byte offset in file)│
  │  footer_length[4]                       │
  │  reserved[18]                           │
  └─────────────────────────────────────────┘

  Segment 0 ... Segment K-1 (variable-length blocks):
  ┌─────────────────────────────────────────┐
  │  SEGMENT S                              │
  │  COL 0 BLOCK:                           │
  │    block_header[16]                     │
  │    encoding_header[0-8]                 │
  │    LZ4-compressed payload               │
  │  COL 1 BLOCK:                           │
  │    ...                                  │
  │  COL N-1 BLOCK                          │
  └─────────────────────────────────────────┘
  (Columns are contiguous within each segment.
   Segments are contiguous in the file.)

  Footer (at EOF - footer_length):
  ┌─────────────────────────────────────────┐
  │  FOOTER HEADER (16 bytes)               │
  │    section_count[4]                     │
  │    section_toc[section_count x 8]       │
  │    (each entry: section_type[2],        │
  │     offset[4], length[4] -- relative    │
  │     to footer start)                    │
  │                                         │
  │  SCHEMA SECTION:                        │
  │    column_count[4]                      │
  │    per-column: name, phys_type,         │
  │      encoding_type, nullable,           │
  │      special_flags (e.g. synthetic_pfx) │
  │                                         │
  │  GLOBAL DICT SECTION:                   │
  │    dict_count[4]                        │
  │    per-dict: col_id[2], entry_count[4], │
  │      entry_bytes[] (LZ4-compressed)     │
  │                                         │
  │  SEGMENT OFFSET TABLE:                  │
  │    num_segments[8]                      │
  │    per-seg: file_byte_offset[8],        │
  │      col_block_offsets[N x 8]           │
  │      (relative to segment start)        │
  │                                         │
  │  ZONE MAP TABLE:                        │
  │    per-seg: per-predicate-col zone-map  │
  │    (min, max, bitset, null_count)       │
  │    (fixed schema, variable per table)   │
  │                                         │
  │  BLOOM FILTER SECTION:                  │
  │    per-seg bloom filters (if any)       │
  │                                         │
  │  AUX METADATA SECTION:                  │
  │    suffix lookup tables, etc.           │
  │                                         │
  │  FOOTER CRC32[4]                        │
  └─────────────────────────────────────────┘

String column block format (encoding_type = RAW_LZ4_STRING):
  Block header as above (16 bytes).
  Additional header:
    offset_block_compressed_size[4]
    payload_block_compressed_size[4]
  Payload (LZ4-compressed):
    offsets[N+1] uint32  (byte boundaries into payload, offsets[0]=0,
                           offsets[N]=total_payload_bytes)
    payload bytes[offsets[N]] (concatenated raw string bytes)
  The offset array and the payload bytes are each LZ4-compressed separately,
  then concatenated.  The split point is given by offset_block_compressed_size.

Null representation (nullable columns only):
  Additional sub-block prepended to the column block.
  Sub-block type: VALIDITY_BITMAP (encoding_type = 0xFE).
  Payload: ceil(N/8) bytes, bit i = 1 means row i is valid (non-null).
  LZ4-compressed.  Non-nullable columns: no sub-block, no overhead.

Bit-packed array format (used by most integer/enum columns):
  Values are packed contiguously, LSB-first within each byte.
  Example: 3 values of 5 bits each in 2 bytes:
    byte 0: bits [0:4]=val0[0:4], bits [5:7]=val1[0:2]
    byte 1: bits [0:1]=val1[3:4], bits [2:6]=val2[0:4], bit 7=padding
  Vectorized unpack uses SIMD pdep/pext or a software fallback.
  All bit-packed arrays are LZ4-compressed after bit-packing.

Reader contract (per-query execution):
  1. On file open: read header (64 B) from offset 0.
     Seek to footer_offset; read footer into RAM.  Pin footer permanently.
  2. For a query with predicate columns P and projected columns C:
     a. For each predicate column: look up its zone-map array in the footer.
     b. Iterate over all segments in order:
        - Evaluate zone maps / bitsets / Bloom filters for each predicate column.
        - If ANY predicate certainly fails (zone_max < low_bound, or bit absent):
          mark segment as SKIP.  Advance to next segment.
        - Else: add segment to READ LIST.
     c. For each segment in READ LIST (in ascending file-offset order):
        i.   Look up per-column block offsets in the segment offset table.
        ii.  Seek to first predicate-column block offset.
        iii. Read and decompress each predicate-column block sequentially.
        iv.  Apply predicate -> produce row-validity bitmap.
        v.   If bitmap is all-zero: skip remaining columns for this segment.
        vi.  For each column in C that is NOT already decoded:
             Seek to that column's block offset; read and decompress.
        vii. Gather output rows using row-validity bitmap (late materialization).
  3. Assemble and return result set.
  All seeks within a segment are short sequential reads (columns packed
  consecutively).  Seeks across pruned segments are coarse (skip MBs of data).
  SSD random-read latency (~100 us) is negligible compared to the MB-scale
  reads per segment.
