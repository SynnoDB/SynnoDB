# Context

You will implement a specialized high-performance database file format in C++ that is optimized for a predefined set of SQL queries. The system has two phases:

1. **Write-to-fileformat phase**: Convert ArrowTable inputs into a workload-specialized custom on-disk file format. The writer should persist compact column representations, encodings, page/block metadata, pruning structures, dictionaries, offsets, and any other file-format structures needed for efficient later reads. Datatypes, encodings, metadata, and layout decisions can be hard-coded for the given workload.

2. **Read-from-fileformat phase**: Execute the predefined queries by reading from the custom file format and writing query results to CSV files. The reader should avoid loading the full dataset into RAM unless explicitly justified. It should use projection pushdown, predicate pushdown, page/block pruning, late materialization, encoded predicate evaluation, and sequential SSD-friendly reads wherever possible.

# System Overview

Queries: Defined in `${queries_path}` (`${query_str}`).

This is a two-phase, loader -> writer -> query pipeline. The C++ interfaces you implement are declared in the read-only BFF API headers in your workspace: `read_api.hpp`, `write_api.hpp`, `ingest_types.hpp`, and `filter_pushdown.hpp`. Do not modify these headers; implement against them.

   Implement `write_bff_from_parquet_tables(const ParquetTables*, ...)` (declared in `write_api.hpp`): encode the already-loaded in-memory Arrow tables (their fields are declared in `parquet_reader.hpp`) into your bespoke on-disk `.bff` layout under the dataset directory. Declare the concrete file-format handles (`BffDataset`, `BffTable`, `BffFooter`) and any layout structs in `bff_format.hpp`. Use an efficient workload-specialized physical representation. `${storage_hint}`

2. **Read-from-fileformat phase** — implemented in `read_impl.cpp` and the per-query files.
   Implement the read API declared in `read_api.hpp` / `filter_pushdown.hpp`: open the dataset, load (and cache) the footer, prune row groups/pages/columns from footer stats, and return byte buffers for the selected ranges. The per-query execution interface is predefined in `${query_impl_path}`, which accepts a `QueryRequest` list and dispatches each request to a `run_q<N>(...)` implementation in its own `query<N>.cpp`. Implement each query by reading from your file format through the read API — using projection pushdown, predicate pushdown, page/block pruning, late materialization, encoded-predicate evaluation, and sequential SSD-friendly reads wherever possible.

Sample query implementations are provided in e.g., `query1.cpp/hpp`. The `Database*` argument is always null for this use-case; the queryable state is the on-disk `.bff` dataset, located via the read API.

The output of each query should be written to `result<RUN_NR>.csv`, where `<RUN_NR>` is the 1-based position in the arg list, using:

```text
delimiter=','
escapechar='\\'
quotechar='"'
header=True
```

Table schemas can be inspected with:


The plan must cover both phases:

   * Decide the physical representation for each required table and column.
